# ColumnLab 系统需求规格说明

> 文档编号：CL-REQ-001  
> 状态：已定稿  
> 优先级定义：P0 必须完成；P1 应完成且默认纳入验收；P2 为时间允许时增强。

---

## 1. 项目背景与目标

ColumnLab 用于展示并验证列式数据库的关键技术：按列存储、块级组织、RLE 与字典编码、多列联合查询、压缩数据直接计算，以及压缩率和查询性能分析。

系统目标不是制作一个上传 CSV 后调用 Pandas 的普通数据分析平台，而是实现一个具有真实物理格式、查询计划和可观测执行过程的单机列式分析引擎。

### 1.1 核心目标

- 设计并实现自定义二进制列式文件格式。
- 设计共享行组和目标字节大小分块算法。
- 实现 RAW、RLE、Dictionary + bit packing 编码。
- 实现块级自适应编码选择。
- 实现 SQL 子集解析、逻辑计划、规则优化、物理执行。
- 实现压缩态过滤、压缩态聚合和局部解码。
- 实现查询、存储和压缩过程的可视化观测。
- 完成自动化正确性测试、故障测试和性能实验。

### 1.2 非目标

下列能力不属于本期范围，不得因此挤压核心实现：

- 分布式存储和分布式查询；
- 多副本、高可用、共识协议；
- 完整事务、MVCC 和 WAL；
- 行级增删改；
- 多表 JOIN；
- 子查询、窗口函数、存储过程；
- 商业级 SQL 兼容；
- 移动端适配；
- 复杂数据清洗、机器学习和零售业务分析。

## 2. 用户与使用场景

### 2.1 系统用户

系统采用多用户模型。每个用户只能访问自己创建或拥有的数据集、导入任务、查询记录和性能实验。

### 2.2 核心使用流程

#### 流程 A：导入数据

1. 用户登录系统。
2. 用户上传 CSV 或 XLSX。
3. 系统以流式方式接收文件并计算 SHA-256。
4. 系统进入异步导入任务。
5. 系统完成格式解析、类型推断、异常值处理策略确认、共享行组切分、编码候选比较、列文件写入、校验与原子提交。
6. 用户实时查看导入阶段、完成比例、处理行数、当前列和错误信息。
7. 导入完成后进入“存储映射”页面。

#### 流程 B：观察列式存储

1. 用户选择数据集与表。
2. 系统展示列清单、逻辑类型、块数量和总体压缩率。
3. 中央区域展示各列共享行组中的物理列块。
4. 用户选择某块。
5. 右侧检查器展示块头、统计信息、CRC、实际编码结果及候选编码比较。
6. 用户可查看字典内容、RLE 游程摘要、空值位图摘要和原始值预览。

#### 流程 C：执行查询

1. 用户输入 SQL 或使用可视化查询构造器。
2. 系统解析并展示逻辑计划。
3. 优化器执行投影裁剪、谓词下推、块裁剪等规则。
4. 系统生成物理计划。
5. 查询执行时，存储映射实时标记未访问、跳过、读取、压缩态计算、局部解码和命中块。
6. 用户查看结果表、执行计划、每算子耗时、读取字节数和解码比例。

#### 流程 D：压缩实验

1. 用户选择列、块或合成数据分布。
2. 系统分别执行 RAW、RLE、Dictionary 编码。
3. 系统记录实际字节数、编码时间、解码时间、过滤时间、聚合时间和峰值内存。
4. 系统展示算法适用边界及最终选择原因。

#### 流程 E：性能基准

1. 用户配置数据规模、基数、连续重复率、块大小、查询类型、预热次数和重复次数。
2. 系统生成可重复实验。
3. 系统输出均值、中位数、P95、标准差、吞吐量、压缩率、裁剪率和解码率。
4. 用户导出 CSV/PNG/JSON 实验结果。

## 3. 功能需求

## 3.1 用户与权限

### FR-AUTH-001（P0）注册与登录

系统应支持用户名、邮箱和密码注册，支持用户名和密码登录，并使用安全密码哈希。

### FR-AUTH-002（P0）会话认证

系统应通过 JWT 或等效机制认证 API，请求不得仅依赖前端传入用户 ID。

### FR-AUTH-003（P0）数据隔离

数据集、导入任务、查询和基准实验必须以用户为边界校验所有权。

---

## 3.2 数据导入

### FR-IMP-001（P0）文件类型

首期必须支持：

- CSV：一等支持；
- XLSX：使用只读流式模式支持。

不要求支持旧版 XLS。

### FR-IMP-002（P0）流式接收

后端不得使用一次性 `await file.read()` 读取完整文件。上传过程必须分块写入临时文件，并同步计算 SHA-256。

默认最大文件大小为 1 GiB，允许通过配置修改。

### FR-IMP-003（P0）异步任务

上传完成后应创建持久化 `ImportJob`，导入不得占用单个 HTTP 请求直到全部完成。

任务状态：

- `pending`
- `parsing`
- `inferring`
- `partitioning`
- `encoding`
- `writing`
- `validating`
- `committing`
- `completed`
- `failed`
- `cancelled`

### FR-IMP-004（P0）类型推断

系统必须支持：

- `INT64`
- `FLOAT64`
- `BOOLEAN`
- `UTF8`
- `DATE32`
- `TIMESTAMP64`
- `DECIMAL64(scale)`

推断过程先采样，再对全量数据验证。用户可在导入前覆盖推断结果。

### FR-IMP-005（P0）异常值策略

导入模式包括：

- 严格模式：任一无法转换的值导致任务失败；
- 宽松模式：无法转换值置为 NULL，并生成错误样本与计数。

不得静默丢弃行。

### FR-IMP-006（P0）共享行组切分

所有列必须共享相同行范围的逻辑行组，以保证多列查询对齐。

行组关闭条件：

- 任意一列的预计未压缩序列化大小达到目标块大小；
- 达到最大行数；
- 输入结束。

默认目标块大小 64 KiB，可配置为 16、32、64、128、256 KiB。默认最小行数 512，最大行数 65,536。

超大单值允许形成单行溢出块，并在块标志中记录。

### FR-IMP-007（P0）原子提交

导入过程先写入临时版本目录。只有完成全部列文件、索引、CRC 和一致性验证后，才能通过原子重命名发布为可见版本。

失败任务不得留下“部分成功”的可查询数据集。

### FR-IMP-008（P1）取消任务

用户可以取消未完成导入。系统应在安全检查点停止并清理临时目录。

## 3.3 列式存储

### FR-STO-001（P0）独立列文件

每个表的每一列必须有独立二进制列文件。

禁止以下实现作为正式存储：

- JSON 数组；
- CSV；
- Pickle；
- NumPy `.npy` 直接代替自定义格式；
- Parquet/ORC 直接代替自定义格式；
- SQLite BLOB 直接保存整列；
- DataFrame 序列化。

### FR-STO-002（P0）文件结构

列文件至少包含：

- File Header；
- 多个 Block Header；
- Block Payload；
- Block Index；
- File Footer；
- Header 与 Payload 校验信息。

### FR-STO-003（P0）块元数据

每个块至少记录：

- block_id；
- row_start；
- row_count；
- logical_type；
- encoding；
- flags；
- null_count；
- distinct_count；
- uncompressed_bytes；
- encoded_bytes；
- min_value；
- max_value；
- dictionary_count 或 run_count；
- payload_offset；
- payload_crc32。

### FR-STO-004（P0）空值

每块使用独立 NULL bitmap。空值不得通过特殊业务值伪装。

### FR-STO-005（P0）索引与裁剪

文件打开时应能够只读取 Footer/Index，获得块行范围、偏移和 min/max，不得为块裁剪先读取全部块载荷。

### FR-STO-006（P0）版本兼容

文件头必须包含魔数和格式版本。未知高版本文件应拒绝读取，并返回明确错误。

### FR-STO-007（P0）损坏检测

读取块前或后应验证 CRC32。损坏块不得返回不可信结果。

### FR-STO-008（P1）块缓存

实现按 `(dataset_version, column_id, block_id)` 标识的线程安全 LRU 块缓存，至少记录命中、未命中、淘汰和占用字节数。

## 3.4 压缩编码

### FR-CMP-001（P0）RAW

实现固定宽度和变长数据的原始二进制编码：

- 数值使用小端固定宽度；
- BOOLEAN 位打包；
- UTF8 使用 offset array + byte blob；
- DATE32 使用天数整数；
- TIMESTAMP64 使用微秒整数；
- DECIMAL64 使用带 scale 的整数。

### FR-CMP-002（P0）RLE

RLE 应保存 `(value, run_length)`。run length 使用无符号变长整数或紧凑整数编码。

必须实现：

- 编码；
- 解码；
- 等值过滤；
- 范围过滤；
- COUNT；
- SUM；
- MIN；
- MAX；
- AVG 的 SUM/COUNT 部分。

以上算子不得先完整展开游程。

### FR-CMP-003（P0）Dictionary

字典编码必须包含：

- 去重字典；
- value → code 映射；
- code 位宽计算；
- codes bit packing；
- 字典值排序/映射策略记录；
- NULL bitmap。

必须实现：

- 编码；
- 解码；
- 等值过滤；
- `IN`；
- 字典快速否定；
- GROUP BY code；
- COUNT；
- 可安全执行时的 MIN/MAX。

### FR-CMP-004（P0）候选编码比较

每个块必须至少生成 RAW、RLE、Dictionary 三个实际候选结果，再根据实际字节数选择，不得只根据列类型写死算法。

若最优候选相对 RAW 的收益低于默认 5%，选择 RAW。

候选差异小于 2% 时，采用较低解码成本的编码，选择规则必须确定且可测试。

### FR-CMP-005（P0）编码解释

系统必须保存候选编码的：

- 实际字节数；
- 压缩收益；
- 编码耗时；
- 选择结果；
- 选择原因。

这些数据必须显示在块检查器中。

### FR-CMP-006（P1）编码策略

支持至少两种策略：

- `size_first`：按最终字节数优先；
- `balanced`：在相近大小时偏向较低查询/解码成本。

默认 `balanced`。

## 3.5 查询语言与计划

### FR-QRY-001（P0）SQL 子集

必须支持单表查询：

```sql
SELECT [列 | 聚合函数 | 别名]
FROM 表
[WHERE 条件]
[GROUP BY 列]
[ORDER BY 列 ASC|DESC]
[LIMIT n [OFFSET m]]
```

条件必须支持：

- `= != > >= < <=`
- `AND OR NOT`
- `IN`
- `BETWEEN`
- `IS NULL`
- 括号

聚合必须支持：

- `COUNT`
- `SUM`
- `AVG`
- `MIN`
- `MAX`

不支持项必须返回明确错误，不得静默改写。

### FR-QRY-002（P0）AST 与逻辑计划

允许使用 SQL 解析库生成 AST，但必须自行构造并展示：

- Scan；
- Filter；
- Project；
- Aggregate；
- Sort；
- Limit。

### FR-QRY-003（P0）规则优化

优化器必须至少实现：

- 投影裁剪；
- 谓词规范化；
- 谓词下推；
- min/max 块裁剪；
- NULL 统计裁剪；
- 字典快速否定；
- LIMIT 提前终止；
- 聚合下推；
- 常量表达式折叠。

### FR-QRY-004（P0）物理计划

至少包含：

- BlockScan；
- RawFilter；
- RLEFilter；
- DictionaryFilter；
- BitmapAnd；
- BitmapOr；
- Materialize；
- HashAggregate；
- Sort；
- Limit。

### FR-QRY-005（P0）多列行对齐

多列过滤通过共享行组和块内 selection bitmap/vector 对齐。不得依赖 Pandas 索引完成交集或并集。

### FR-QRY-006（P0）压缩态执行

能够在编码表示上完成的操作必须使用专用算子。只有无法安全直接执行的操作才允许局部解码。

查询统计应区分：

- 压缩态处理块数；
- 局部解码块数；
- 完整物化行数。

### FR-QRY-007（P0）执行指标

每次查询至少记录：

- parse_time；
- optimize_time；
- execute_time；
- total_time；
- total_blocks；
- pruned_blocks；
- scanned_blocks；
- cache_hits；
- bytes_read；
- rows_examined；
- rows_output；
- compressed_operator_blocks；
- decoded_blocks；
- peak_memory；
- 各物理算子输入/输出行数与耗时。

### FR-QRY-008（P1）取消查询

长查询应支持取消，取消后释放读取缓冲和临时结果。

## 3.6 压缩与性能实验

### FR-BEN-001（P0）合成数据生成器

至少支持以下数据分布：

- 长连续游程；
- 低基数随机；
- 高基数；
- 均匀数值；
- 偏斜数值；
- 含 NULL；
- 混合业务表。

生成参数必须可保存，并支持固定随机种子。

### FR-BEN-002（P0）算法对比

对 RAW、RLE、Dictionary 测量：

- 编码字节数；
- 压缩率；
- 编码时间；
- 解码时间；
- 等值过滤时间；
- 范围过滤时间；
- 聚合时间；
- 峰值内存。

### FR-BEN-003（P0）查询实验

至少支持：

- 行式基线与列式查询；
- 无裁剪与块裁剪；
- 完整解码与压缩态执行；
- 冷缓存与热缓存；
- 不同目标块大小；
- 不同数据规模。

### FR-BEN-004（P0）统计方法

每项实验应支持：

- 预热次数；
- 重复次数；
- 均值；
- 中位数；
- P95；
- 标准差；
- 原始样本保存。

禁止只展示一次运行时间。

## 4. 非功能需求

### NFR-001 正确性（P0）

- 所有编码满足 `decode(encode(values)) == values`，包括 NULL。
- SQL 子集结果与 DuckDB/Pandas 参照结果一致。
- 相同输入、策略和版本生成确定性结果。

### NFR-002 可靠性（P0）

- 导入中断不暴露半成品；
- CRC 错误可定位到具体文件和块；
- 任务失败保留结构化错误，不只记录字符串日志；
- 删除数据集同时删除列文件、任务和历史，但操作需二次确认。

### NFR-003 性能（P0）

性能验收以相对指标和合成数据为主：

- 选择性范围查询应实际跳过符合预期的块；
- 低基数字符串字典编码应显著小于 RAW；
- 长游程数据 RLE 应显著小于 RAW；
- 查询只读取涉及列；
- 不允许查询时默认加载整表 DataFrame。

### NFR-004 可维护性（P0）

- `storage_engine`、`codecs`、`query_engine` 不得依赖 FastAPI；
- 核心算法可通过纯 Python 测试直接调用；
- 每个公共类和文件格式字段有文档；
- 不允许单文件实现全部引擎。

### NFR-005 可测试性（P0）

- pytest；
- Hypothesis 属性测试；
- 文件格式 golden test；
- 损坏文件测试；
- API 集成测试；
- 前端关键流程 E2E；
- 性能基准不作为普通单元测试阻塞，但必须可一键运行。

### NFR-006 安全（P0）

- 文件名不直接用于物理路径；
- 防止路径穿越；
- 上传大小和类型校验；
- 所有权校验；
- SQL 只解析白名单子集；
- 错误响应不得暴露服务器绝对路径。

### NFR-007 可观测性（P1）

- 结构化日志；

## 5. 数据与目录约束

推荐物理目录：

```text
storage/
└── tenants/{user_id}/
    └── datasets/{dataset_id}/
        └── versions/{version_id}/
            ├── manifest.bin
            ├── schema.bin
            └── tables/{table_id}/
                ├── table.meta
                └── columns/{column_id}.col
```

临时导入：

```text
storage/.staging/{job_id}/
```

不得使用用户提供的原始文件名构造最终路径。

## 6. 必须禁止的简化实现

以下任一项出现，视为核心实现不合格：

1. 用 Pandas 完成正式查询执行。
2. 查询前完整解压整列或整表。
3. 用 JSON 保存列块或块头。
4. 用固定行数冒充目标字节分块，且没有字节预算。
5. 各列独立分块而没有可靠的全局行号对齐机制。
6. 仅实现 RLE/字典函数，却未接入存储和查询。
7. 字典 codes 使用 Python int 列表直接落盘，不做位宽控制或 bit packing。
8. 只根据字符串/数值类型写死压缩算法。
9. 用 Parquet、DuckDB 或 SQLite 替代核心列式引擎。
10. 前端展示固定压缩率、固定耗时或模拟执行计划。
11. 只实现成功路径，没有 CRC、回滚和异常状态。
12. 复用 RetailInsight 的黄绿主题或传统 Dashboard 布局。
